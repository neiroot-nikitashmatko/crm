# Памятка по базе данных PROclients

Простое объяснение: **что хранится в PostgreSQL, как таблицы связаны и как это работает в приложении.**

---

## 1. Что это вообще такое

**База данных (БД)** — это одно большое хранилище на сервере, куда программа записывает все важные данные: клиентов, сделки, задачи, файлы и т.д.

В нашем проекте используется **PostgreSQL**. Приложение **не пишет в БД напрямую из браузера**. Цепочка такая:

```
Браузер (Vue)  →  Go API (backend)  →  PostgreSQL
```

Все изменения идут через API. БД — единственный источник правды для постоянных данных.

Подключение к БД настраивается в `backend/.env` через переменную **`DATABASE_URL`**.

---

## 2. Главная идея (как устроен бизнес)

Представь воронку продаж:

1. **Лид** — потенциальный клиент (только заявка, ещё не сделка).
2. **Сделка** — когда лид «созрел», из него делают сделку.
3. **Задачи** — напоминания менеджеру (позвонить, уточнить и т.п.).
4. **Товары** — что продаём (каталог + строки в лиде/сделке).
5. **Производство** — что делаем, когда и кто делает.
6. **Самовывоз / доставка** — адреса и даты.
7. **Вложения** — файлы (фото, документы).
8. **Активности** — лента: комментарии и системные события («сделка создана», «файл загружен»).



## 3. Все таблицы (кратко)

| Таблица | Зачем нужна |
|---------|-------------|
| `users` | Сотрудники: логин, пароль, роль (админ/менеджер) |
| `leads` | Лиды (канбан лидов) |
| `lead_products` | Товары/услуги в лиде (до создания сделки) |
| `deals` | Сделки (канбан сделок) |
| `deal_products` | Товары/услуги в сделке |
| `tasks` | Задачи по лиду или сделке |
| `catalog_products` | Справочник товаров (название, артикул, цена) |
| `attachments` | Файлы к сделкам и задачам (хранятся в БД) |
| `activities` | Лента событий и комментариев |

**Всего 9 таблиц** (на момент миграции 017).

## 4. Таблицы подробно

### 4.1. `users` — сотрудники

Кто может войти в систему.

| Поле | Что это |
|------|---------|
| `id` | Уникальный номер записи (UUID) |
| `first_name`, `last_name`, `patronymic` | ФИО |
| `phone` | Телефон для входа (формат `+79001234567`) |
| `password_hash` | Пароль в зашифрованном виде (не храним пароль открытым текстом) |
| `role` | `admin` или `manager` |
| `position` | Должность |
| `birth_date` | Дата рождения |
| `is_active` | Активен ли сотрудник (`false` = «удалён», но запись остаётся) |
| `created_at`, `updated_at` | Когда создан / обновлён |

---

### 4.2. `leads` — лиды

Карточка потенциального клиента на канбане лидов.

| Поле | Что это |
|------|---------|
| `id` | UUID |
| `lead_number` | Порядковый номер лида (автоматически: 1, 2, 3…) |
| `first_name`, `patronymic`, `phone` | Контакт клиента |
| `traffic_source` | Откуда пришёл (реклама, сайт и т.д.) |
| `column_id` | Колонка канбана: `new`, `chat`, `phone`, `deal`, `failed` |
| `lead_comments` | Комментарий по лиду |
| `failure_reason` | Причина провала (если лид в колонке `failed`) |
| `production_nomenclature` | Что производим (до сделки) |
| `production_due_at` | Когда нужно сделать |
| `production_employee` | Кто делает на производстве |
| `pickup_address`, `pickup_date` | Самовывоз |
| `delivery_address`, `delivery_date`, `courier` | Доставка |
| `created_by` | Кто создал (ссылка на `users.id`) |
| `deleted_at` | Если заполнено — лид «удалён» (мягкое удаление) |
| `created_at`, `updated_at` | Даты |

**Товары лида** лежат в отдельной таблице `lead_products` (см. ниже).

---

### 4.3. `lead_products` — товары в лиде

Строки «Услуги/Товары» в карточке лида (пока сделки ещё нет).

| Поле | Что это |
|------|---------|
| `lead_id` | К какому лиду относится |
| `position` | Порядок строки (0, 1, 2…) |
| `title` | Название товара/услуги |
| `quantity` | Количество |
| `unit_price` | Цена за единицу |
| `amount` | Сумма строки (количество × цена) |

При удалении лида строки удаляются автоматически (`ON DELETE CASCADE`).

---

### 4.4. `deals` — сделки

Основная рабочая сущность после конвертации лида.

| Поле | Что это |
|------|---------|
| `id` | UUID |
| `deal_number` | Порядковый номер сделки |
| `lead_id` | Из какого лида создана (может быть пусто у старых записей) |
| `first_name`, `patronymic`, `phone`, `traffic_source` | Данные клиента (копируются из лида) |
| `status` | Статус канбана сделок: `today`, `tomorrow`, `later`, `closed`, `failed` |
| `total_amount` | Итоговая сумма по товарам |
| `deal_comments` | Комментарий |
| `failure_reason` | Причина провала |
| `production_nomenclature`, `production_due_at`, `production_employee` | Производство |
| `pickup_*`, `delivery_*`, `courier` | Самовывоз и доставка |
| `created_by` | Кто создал |
| `deleted_at` | Мягкое удаление |
| `created_at`, `updated_at` | Даты |

**Важное правило:** у одного лида может быть **только одна активная сделка** (не проваленная и не удалённая). Это контролируется индексом `uq_deals_active_per_lead`.

**Товары сделки** — в таблице `deal_products`.

---

### 4.5. `deal_products` — товары в сделке

То же, что `lead_products`, но привязано к сделке.

| Поле | Что это |
|------|---------|
| `deal_id` | К какой сделке |
| `position`, `title`, `quantity`, `unit_price`, `amount` | Как в `lead_products` |

При сохранении товаров backend пересчитывает `deals.total_amount`.

---

### 4.6. `tasks` — задачи

Напоминания: позвонить, уточнить, встретиться.

| Поле | Что это |
|------|---------|
| `title`, `text` | Заголовок и описание |
| `due_at` | Срок (дата и время) |
| `status` | `active` или `completed` |
| `lead_id` | Задача по лиду (одно из двух обязательно) |
| `deal_id` | Задача по сделке |
| `created_by` | Кто создал |
| `completed_at` | Когда завершили |
| `created_at`, `updated_at` | Даты |

Задача **обязана** быть привязана хотя бы к лиду **или** к сделке.

---

### 4.7. `catalog_products` — каталог товаров

Справочник: из него в UI выбирают товар при добавлении строки в лид/сделку.

| Поле | Что это |
|------|---------|
| `name` | Название |
| `sku` | Артикул (уникальный) |
| `category` | Категория |
| `cost` | Базовая цена из каталога |

В `lead_products` / `deal_products` сохраняется **копия** названия и цены на момент добавления (можно изменить вручную в карточке).

---

### 4.8. `attachments` — вложения (файлы)

Файлы прикрепляются к **сделке** или **задаче**.

| Поле | Что это |
|------|---------|
| `entity_type` | `deal` или `task` |
| `entity_id` | ID сделки или задачи |
| `name` | Имя файла |
| `size` | Размер в байтах |
| `mime_type` | Тип (например `image/jpeg`) |
| `content` | Сам файл в бинарном виде (`BYTEA`) |
| `uploaded_by` | Кто загрузил |
| `deleted_at` | Мягкое удаление |

Файлы лежат **внутри PostgreSQL**, не на диске отдельно.

---

### 4.9. `activities` — лента активностей

История в правой колонке карточки сделки/задачи.

| Поле | Что это |
|------|---------|
| `entity_type` | `deal` или `task` |
| `entity_id` | ID сущности |
| `activity_type` | `system` (автоматически) или `comment` (комментарий пользователя) |
| `text` | Текст события |
| `author_id` | Кто написал / от чьего имени система |
| `deleted_at` | Мягкое удаление |
| `created_at` | Когда произошло |

Примеры системных записей: «Сделка создана», «Дата производства изменена», «Загружено 2 файла».

## 5. Как связаны таблицы (простыми словами)

| Откуда | Куда | Связь |
|--------|------|--------|
| `leads.created_by` | `users.id` | Лид создал сотрудник |
| `deals.lead_id` | `leads.id` | Сделка из лида |
| `deals.created_by` | `users.id` | Сделку создал сотрудник |
| `lead_products.lead_id` | `leads.id` | Товары лида |
| `deal_products.deal_id` | `deals.id` | Товары сделки |
| `tasks.lead_id` | `leads.id` | Задача по лиду |
| `tasks.deal_id` | `deals.id` | Задача по сделке |
| `tasks.created_by` | `users.id` | Задачу создал сотрудник |
| `attachments` | deal/task по `entity_type` + `entity_id` | Файл к сделке или задаче |
| `activities` | deal/task по `entity_type` + `entity_id` | Событие в ленте |

`catalog_products` **не связан** внешним ключом с лидами/сделками — это просто справочник для выбора в интерфейсе.

## 6. Статусы и колонки (что значат значения)

### Лиды — `leads.column_id`

| Значение | Колонка в UI |
|----------|----------------|
| `new` | Новые |
| `chat` | Переписка |
| `phone` | Звонок |
| `deal` | Сделка (лид конвертирован) |
| `failed` | Провал |

### Сделки — `deals.status`

| Значение | Колонка в UI |
|----------|----------------|
| `today` | Производство (сегодня) |
| `tomorrow` | Самовывоз |
| `later` | Доставка |
| `closed` | Закрытые |
| `failed` | Проваленные |

### Задачи — `tasks.status`

| Значение | Смысл |
|----------|--------|
| `active` | Активная |
| `completed` | Завершена |

### Роли — `users.role`

| Значение | Смысл |
|----------|--------|
| `admin` | Полный доступ, раздел «Сотрудники» |
| `manager` | Обычный пользователь CRM |

## 7. Мягкое удаление

У **лидов** и **сделок** есть поле `deleted_at`:

- Пустое (`NULL`) — запись **живая**, видна в приложении.
- Заполнено датой — запись **удалена**, в списках не показывается, но в БД остаётся.

То же для `attachments` и `activities` — файл/запись ленты можно «удалить», не стирая физически сразу.

У **сотрудников** вместо `deleted_at` используется `is_active = false`.

## 8. Типы данных (коротко)

| Тип в БД | Что хранит |
|----------|------------|
| `UUID` | Длинный уникальный id (например `a1b2c3d4-...`) |
| `TIMESTAMPTZ` | Дата и время с часовым поясом |
| `TEXT` | Текст любой длины |
| `NUMERIC(12,2)` | Деньги (до копеек) |
| `BYTEA` | Бинарные данные (файлы) |
| `BOOLEAN` | Да/нет |
| `BIGINT` | Большое целое (номера лидов/сделок) |

В API даты часто уходят на фронт как **миллисекунды** (timestamp в JS), в БД — как `TIMESTAMPTZ`.

## 9. Миграции — как обновлять схему на сервере

Схема БД меняется **только через SQL-файлы** в папке:

```
backend/migrations/
```

Файлы идут по номерам: `001_...`, `002_...`, … `017_...`.

| Что | Где |
|-----|-----|
| Применить миграции (локально, Docker) | `./backend/scripts/apply_migrations.sh` |
| Откат одной миграции | файл `NNN_имя.down.sql` (вручную, осторожно) |
| Создать админа | `backend/scripts/create_admin.sql.example` |

**На продакшене** перед первым запуском приложения нужно:

1. Создать базу PostgreSQL.
2. Прописать `DATABASE_URL` в `backend/.env`.
3. Применить все миграции `001` → `017` по порядку.
4. Создать первого администратора (скрипт/SQL, не из репозитория с паролем).

Не править таблицы вручную в prod — только новые миграции.

## 10. Что где лежит в жизненном цикле клиента

| Этап | Таблицы |
|------|---------|
| Менеджер завёл лида | `leads` |
| Написал товары, производство, самовывоз в лиде | `leads` + `lead_products` |
| Создал задачу «перезвонить» | `tasks` (с `lead_id`) |
| Нажал «Создать сделку» | `deals`, `deal_products`, лид → `column_id = deal` |
| Работа по сделке | `deals`, `deal_products`, `tasks`, `attachments`, `activities` |
| Календарь производства | читает `deals` (поля production + статус) |

После создания сделки **источник правды по товарам** — `deal_products`. Данные в лиде могут остаться для истории.
